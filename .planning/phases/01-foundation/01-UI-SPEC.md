# Phase 1 вҖ” UI Design Contract

> Visual and interaction contract for Phase 1: Foundation (options page CRUD + shared CSS).
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

**Scope:** Options page (`options.html` + `options.js`) and the shared stylesheet (`styles/main.css`)
that will also be consumed by the popup in Phase 2. No popup UI, no injection UI вҖ” those are
Phase 2 and 3 respectively.

**Tool stack:** Vanilla HTML/CSS/JS only. No component library, no build step, no external font
load. All values below are expressed as raw CSS custom properties or literal pixel values.

## Color

Expressed as CSS custom properties in `styles/main.css`. Source: CONTEXT.md D-13.

| Role | Value | CSS var | Usage |
|------|-------|---------|-------|
| Dominant (60%) | `#ffffff` | `--color-surface` | Page background, form background, list panel background |
| Secondary (30%) | `#f3f4f6` | `--color-surface-secondary` | List panel bg distinguisher, input backgrounds, hover row tint |
| Border | `#e5e7eb` | `--color-border` | Panel divider, input borders, list item separators |
| Text primary | `#111827` | `--color-text` | All body copy, headings, form values |
| Text muted | `#6b7280` | `--color-text-muted` | Placeholder text, helper copy, category value when deselected |
| Accent (10%) | `#2563eb` | `--color-accent` | See reserved list below |
| Accent hover | `#1d4ed8` | `--color-accent-hover` | Accent element hover state only |
| Destructive | `#dc2626` | `--color-destructive` | Delete confirmation button ("Are you sure?") state only |
| Destructive hover | `#b91c1c` | `--color-destructive-hover` | Destructive button hover state only |

**Accent reserved for (explicit list):**
1. The `[Save Prompt]` button (primary action) вҖ” background fill
2. The `[+ New]` button вҖ” background fill
3. The selected/active list item highlight вҖ” left border stripe (3px) + light tint background (`#eff6ff`)
4. Focused input ring (`outline: 2px solid var(--color-accent)`)

Accent is NOT used on: general links, labels, headings, borders, idle list items, or the `[Delete]` button in its default state.

## Interaction States

### Prompt List Item

| State | Visual |
|-------|--------|
| Default | `background: var(--color-surface)`, full-width row, `padding: var(--space-sm) var(--space-md)` |
| Hover | `background: var(--color-surface-secondary)` |
| Selected/active | Left border stripe `3px solid var(--color-accent)` + `background: #eff6ff` |

### Buttons

| Button | Default | Hover | Disabled |
|--------|---------|-------|----------|
| `[+ New]` | `background: var(--color-accent)`, white text, 6px border-radius | `var(--color-accent-hover)` | `opacity: 0.5; cursor: not-allowed` |
| `[Save Prompt]` | `background: var(--color-accent)`, white text, 6px border-radius | `var(--color-accent-hover)` | `opacity: 0.5; cursor: not-allowed` |
| `[Delete]` (default) | `background: transparent`, `border: 1px solid var(--color-border)`, muted text | `background: var(--color-surface-secondary)` | вҖ” |
| `[Are you sure?]` (armed) | `background: var(--color-destructive)`, white text | `var(--color-destructive-hover)` | вҖ” |

### Delete Two-Step Confirmation (D-11)

1. User clicks `[Delete]`: button label changes to `[Are you sure?]`, button style changes to destructive fill
2. User clicks `[Are you sure?]`: deletion executes, list updates, form clears
3. User clicks anywhere else while button is armed: button resets to `[Delete]` default state
4. No browser `confirm()` dialog вҖ” all inline on the button element itself

### Form Fields

| State | Visual |
|-------|--------|
| Idle | `border: 1px solid var(--color-border)`, `background: var(--color-surface-secondary)`, `border-radius: 4px` |
| Focused | `outline: 2px solid var(--color-accent)`, `outline-offset: 0`, border color unchanged |
| Empty (placeholder) | Placeholder text in `var(--color-text-muted)` |

### `[+ New]` Button Behavior (D-06)

- Clears all form fields (title, category, body)
- Removes active highlight from any selected list item
- Does NOT insert a blank row in the list
- Right panel heading changes to "New Prompt" when form is blank, "Edit Prompt" when editing an existing item

## Sample Prompt Copy Contract

Source: CONTEXT.md D-09, D-10.

Three categories: `Coding`, `Explaining`, `Writing`. Target audience: junior devs,
self-learners, non-native English speakers. No `[PLACEHOLDER]` template style.
Aim for 5вҖ“8 prompts total.

Example tone bar (for executor to calibrate against):
- Good: "Explain this error to me like I'm a junior developer. What caused it and how do I fix it?"
- Avoid: "Act as a [ROLE] expert and analyze [TOPIC] with respect to [CONTEXT]."

Exact sample prompt copy is the executor's responsibility (matching the tone bar above and the
distribution guideline: 2вҖ“3 Coding, 2 Explaining, 1вҖ“2 Writing). The UI-SPEC does not prescribe
exact bodies to allow the executor to write prompts that feel natural.

## CSS Architecture

Source: CONTEXT.md D-12, D-13.

- Single shared stylesheet: `styles/main.css`
- Imported by `options.html` via `<link rel="stylesheet" href="styles/main.css">`
- Will be imported by `popup.html` in Phase 2 via the same path
- All CSS custom properties declared in `:root {}` at the top of `main.css`
- No CSS preprocessor, no utility-class framework, no CSS modules
- Selector convention: BEM-lite вҖ” e.g. `.prompt-list`, `.prompt-list__item`, `.prompt-list__item--active`, `.form-panel`, `.btn`, `.btn--primary`, `.btn--destructive`

## Accessibility Baseline

- All form fields have explicit `<label for="...">` associations (no `placeholder` as the only label)
- Delete two-step: armed state must have `aria-pressed="true"` or equivalent so screen readers announce the state change
- Selected list item: `aria-current="true"` on the active `<li>` element
- Buttons use `<button type="button">` (never `<div>` or `<span>` as interactive elements)
- Focus is visible at all times вҖ” the 2px accent outline on focused elements satisfies WCAG 2.1 AA 3.2 minimum
